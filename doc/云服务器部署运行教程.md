# 云服务器部署运行教程

> 更新时间：2026-04-23  
> 当前正式主实验流程：`run_sae_evaluation.py -> causal/run_experiment.py`  
> 当前默认数据集：`data/mi_re`  
> 当前 SAE checkpoint 语义：显式使用 `--checkpoint-topk-semantics hard`

这份文档说明如何把当前项目部署到一台 Linux GPU 云服务器，并跑通正式主实验与因果验证实验。仓库里的云端脚本位于 `deploy/gce/`，虽然目录名叫 `gce`，但只要是 Ubuntu GPU 云服务器也可以使用。

---

## 1. 云端脚本是否已适配当前代码

已适配。当前云端脚本和本地正式流程保持一致：

- `deploy/gce/env.example`
  - 默认 `PYTHON_BIN=python3.11`
  - 默认 `DATA_DIR=data/mi_re`
  - 默认 `DEVICE=cuda`
  - 默认 `CHECKPOINT_TOPK_SEMANTICS=hard`
- `deploy/gce/run_full_eval.sh`
  - 调用 `run_sae_evaluation.py`
  - 显式传入 `--data-dir "${DATA_DIR}"`
  - 显式传入 `--checkpoint-topk-semantics "${CHECKPOINT_TOPK_SEMANTICS}"`
- `deploy/gce/run_causal.sh`
  - 调用 `causal/run_experiment.py`
  - 默认读取 `${SAE_OUTPUT_DIR}/candidate_latents.csv`
  - 显式传入同一套 `DATA_DIR` 和 `CHECKPOINT_TOPK_SEMANTICS`
- `deploy/gce/bootstrap.sh`
  - 使用 Python 3.11 创建虚拟环境
  - 安装 `lm-saes` 和 `transformer-lens`
  - 校验 official SAE backend 可导入

这意味着云端默认不再跑论文代理数据集，也不默认跑 `cactus`。`cactus` 仍然兼容，但需要你主动改 `DATA_DIR=data/cactus`。

---

## 2. 推荐服务器配置

最低建议：

- Ubuntu 22.04 或 24.04
- 1 张 NVIDIA GPU
- 24GB 显存更稳
- 100GB 以上磁盘
- `nvidia-smi` 可用

更稳的配置：

- L4 / A10 / A100
- 200GB 以上磁盘
- 单独挂载数据盘到 `/mnt/disks/data`

先在云服务器上检查：

```bash
nvidia-smi
python3 --version
git --version
```

如果 `nvidia-smi` 不可用，先处理驱动和 CUDA 环境，不要继续跑实验。

## 3. 上传项目

### 方式 A：git clone

```bash
git clone <你的仓库地址> ~/nlp-re
cd ~/nlp-re
```

### 方式 B：本地打包后上传

如果代码还没有推到 GitHub，可以在本地项目根目录执行：

```powershell
python package_project.py
```

把生成的 `tar.gz` 上传到云服务器后解压：

```bash
mkdir -p ~/nlp-re
tar -xzf <你的源码包>.tar.gz -C ~/nlp-re
cd ~/nlp-re
```

## 4. 配置云端环境变量

复制模板：

```bash
cp deploy/gce/env.example deploy/gce/.env
nano deploy/gce/.env
```

至少确认这些项：

```bash
DATA_ROOT=/mnt/disks/data
MODEL_DIR=${DATA_ROOT}/models/Llama-3.1-8B
HF_HOME=${DATA_ROOT}/hf-cache
OUTPUT_ROOT=${DATA_ROOT}/outputs

VENV_DIR=${HOME}/nlp-re/.venv
PYTHON_BIN=python3.11
PYTORCH_INDEX_URL=https://download.pytorch.org/whl/cu124

DATA_DIR=data/mi_re
DEVICE=cuda
CHECKPOINT_TOPK_SEMANTICS=hard

HF_TOKEN=<你的 Hugging Face token>
MODEL_HF_REPO_ID=meta-llama/Llama-3.1-8B
MODEL_REVISION=main

SAE_OUTPUT_DIR=${OUTPUT_ROOT}/sae_eval_full
CAUSAL_OUTPUT_DIR=${OUTPUT_ROOT}/causal_validation_full
```

如果没有挂载 `/mnt/disks/data`，可以改成：

```bash
DATA_ROOT=${HOME}/data
```

当前代码要求 official `lm-saes` backend。请保持 `PYTHON_BIN=python3.11`，否则 `requirements.txt` 中的 `lm-saes` 和 `transformer-lens` 不会安装。

## 5. 初始化环境

执行：

```bash
bash deploy/gce/bootstrap.sh
```

这个脚本会做这些事：

- 安装 Python 3.11、venv、pip、tmux、git
- 创建 `${VENV_DIR}`
- 安装 PyTorch
- 安装 `requirements.txt`
- 执行 `pip install -e .`
- 校验 `torch`、`lm_saes`、`transformer_lens`
- 校验三个入口脚本的 `--help`

成功时会看到：

```text
Bootstrap completed successfully.
```

常见失败点：

- `deploy/gce/.env` 不存在
- Ubuntu 源无法安装 Python 3.11
- 网络无法访问 PyPI 或 PyTorch wheel 源
- GPU 驱动不可用

## 8. 跑 SAE 主实验

前台运行：

```bash
bash deploy/gce/run_full_eval.sh
```

推荐用 tmux 后台运行：

```bash
bash deploy/gce/start_full_eval_tmux.sh
```

查看状态：

```bash
tmux ls
tmux attach -t sae_eval
```

查看日志：

```bash
tail -f ${SAE_OUTPUT_DIR}/run.log
```

实际执行的关键参数等价于：

```bash
python run_sae_evaluation.py \
  --model-dir "${MODEL_DIR}" \
  --device "${DEVICE}" \
  --data-dir "${DATA_DIR}" \
  --output-dir "${SAE_OUTPUT_DIR}" \
  --batch-size "${SAE_BATCH_SIZE}" \
  --max-seq-len "${SAE_MAX_SEQ_LEN}" \
  --checkpoint-topk-semantics "${CHECKPOINT_TOPK_SEMANTICS}" \
  --full-structural
```

成功后重点看：

- `${SAE_OUTPUT_DIR}/metrics_structural.json`
- `${SAE_OUTPUT_DIR}/metrics_ce_kl.json`
- `${SAE_OUTPUT_DIR}/metrics_functional.json`
- `${SAE_OUTPUT_DIR}/candidate_latents.csv`
- `${SAE_OUTPUT_DIR}/judge_bundle/`
- `${SAE_OUTPUT_DIR}/latent_cards/`

## 9. 跑因果验证实验

先确保 SAE 主实验已经生成：

```bash
${SAE_OUTPUT_DIR}/candidate_latents.csv
```

前台运行：

```bash
bash deploy/gce/run_causal.sh
```

推荐用 tmux 后台运行：

```bash
bash deploy/gce/start_causal_tmux.sh
```

查看日志：

```bash
tail -f ${CAUSAL_OUTPUT_DIR}/run.log
```

因果验证现在会额外写出更细的运行状态文件：

```bash
tail -f ${CAUSAL_OUTPUT_DIR}/causal_run.log
cat ${CAUSAL_OUTPUT_DIR}/run_status.json
tail -n 20 ${CAUSAL_OUTPUT_DIR}/run_events.jsonl
cat ${CAUSAL_OUTPUT_DIR}/fatal_traceback.log
```

这些文件的作用：

- `run.log`：`deploy/gce/run_causal.sh` 外层脚本日志，包含 stdout 和 stderr。
- `causal_run.log`：`causal/run_experiment.py` 内部 tee 日志，记录每个阶段和 batch 进度。
- `run_status.json`：当前阶段、耗时、失败阶段、异常类型和 traceback。长时间运行时优先看这个文件。
- `run_events.jsonl`：阶段开始、结束、失败的事件流，适合判断最后完成到了哪一步。
- `fatal_traceback.log`：Python fatal error / native crash 的兜底堆栈；如果程序被系统 OOM kill，这里可能为空，需要再看 `dmesg`。

如果任务看起来“自己停了”，按这个顺序排查：

```bash
cat ${CAUSAL_OUTPUT_DIR}/run_status.json
tail -n 80 ${CAUSAL_OUTPUT_DIR}/causal_run.log
tail -n 80 ${CAUSAL_OUTPUT_DIR}/run.log
dmesg -T | tail -n 80
```

实际执行的关键参数等价于：

```bash
python -u causal/run_experiment.py \
  --model-dir "${MODEL_DIR}" \
  --device "${DEVICE}" \
  --candidate-csv "${SAE_OUTPUT_DIR}/candidate_latents.csv" \
  --data-dir "${DATA_DIR}" \
  --output-dir "${CAUSAL_OUTPUT_DIR}" \
  --batch-size "${CAUSAL_BATCH_SIZE}" \
  --max-seq-len "${CAUSAL_MAX_SEQ_LEN}" \
  --checkpoint-topk-semantics "${CHECKPOINT_TOPK_SEMANTICS}" \
  --lambdas 0.5 1.0 1.5 2.0
```

成功后重点看：

- `${CAUSAL_OUTPUT_DIR}/run_status.json`
- `${CAUSAL_OUTPUT_DIR}/run_events.jsonl`
- `${CAUSAL_OUTPUT_DIR}/causal_run.log`
- `${CAUSAL_OUTPUT_DIR}/fatal_traceback.log`
- `${CAUSAL_OUTPUT_DIR}/selected_groups.json`
- `${CAUSAL_OUTPUT_DIR}/results_necessity.json`
- `${CAUSAL_OUTPUT_DIR}/results_sufficiency.json`
- `${CAUSAL_OUTPUT_DIR}/results_selectivity.json`
- `${CAUSAL_OUTPUT_DIR}/results_group.json`
- `${CAUSAL_OUTPUT_DIR}/summary_tables.md`

## 10. 可选：跑 AI 专家代理评审

如果要跑 AI judge，在 `.env` 中补充：

```bash
OPENAI_API_KEY=<你的 API key>
OPENAI_BASE_URL=https://dashscope.aliyuncs.com/compatible-mode/v1
OPENAI_MODEL=qwen3.5-plus
OPENAI_EXTRA_BODY_JSON='{"enable_thinking":false}'

JUDGE_TOP_LATENTS=20
JUDGE_TOP_N=10
JUDGE_CONTROL_N=5
JUDGE_GROUPS=G1,G5,G20
JUDGE_REQUEST_TIMEOUT=600
```

前台运行：

```bash
bash deploy/gce/run_ai_judge.sh
```

后台运行：

```bash
bash deploy/gce/start_ai_judge_tmux.sh
```

默认输入：

```bash
${SAE_OUTPUT_DIR}
```

默认输出：

```bash
${SAE_OUTPUT_DIR}/ai_judge
```

如果你使用其他兼容 OpenAI API 的服务，只要设置对应的 `OPENAI_BASE_URL`、`OPENAI_MODEL` 和必要的 `OPENAI_EXTRA_BODY_JSON` 即可。

## 12. 如何切换数据集

当前正式默认是：

```bash
DATA_DIR=data/mi_re
```

如果你明确要跑兼容的 CACTUS 小数据集，修改 `.env`：

```bash
DATA_DIR=data/cactus
```

或者临时覆盖：

```bash
bash deploy/gce/run_full_eval.sh --data-dir data/cactus
bash deploy/gce/run_causal.sh --data-dir data/cactus
```

注意：正式报告口径当前以 `data/mi_re` 为准，`cactus` 不再是默认主流程。

## 13. 常见问题

### `lm-saes is not installed`

通常是 Python 版本不对。确认：

```bash
grep PYTHON_BIN deploy/gce/.env
```

应为：

```bash
PYTHON_BIN=python3.11
```

然后重新执行：

```bash
bash deploy/gce/bootstrap.sh
```

### `HF_TOKEN is empty`

说明 `.env` 里还没有填 Hugging Face token。填好后重新跑：

```bash
bash deploy/gce/download_model.sh
```

### `Local model path not found`

说明 `${MODEL_DIR}` 不存在或模型没下载完整。检查：

```bash
echo "$MODEL_DIR"
ls -lah "$MODEL_DIR"
```

必要时重新执行：

```bash
bash deploy/gce/download_model.sh
```

### `Candidate CSV not found`

说明因果实验启动时没有找到：

```bash
${SAE_OUTPUT_DIR}/candidate_latents.csv
```

先跑完：

```bash
bash deploy/gce/run_full_eval.sh
```

### CUDA 显存不足

优先在 `.env` 里调小：

```bash
SAE_BATCH_SIZE=2
CAUSAL_BATCH_SIZE=2
SAE_MAX_SEQ_LEN=128
CAUSAL_MAX_SEQ_LEN=128
```

如果仍然不足，可以继续把 batch size 调到 `1`。

### SSH 断开

长任务不要直接在普通 SSH 前台跑。使用：

```bash
tmux ls
tmux attach -t sae_eval
tmux attach -t causal_eval
tmux attach -t ai_judge
```

## 14. 一句话总结

云服务器上跑通当前实验就是这几步：

1. 上传代码
2. 填好 `deploy/gce/.env`
3. 执行 `bootstrap.sh`
4. 执行 `download_model.sh`
5. 用 `start_full_eval_tmux.sh` 跑 SAE 主实验
6. 用 `start_causal_tmux.sh` 跑因果验证
7. 可选用 `start_ai_judge_tmux.sh` 跑 AI 专家代理评审

最重要的三个口径是：

- `PYTHON_BIN=python3.11`
- `DATA_DIR=data/mi_re`
- `CHECKPOINT_TOPK_SEMANTICS=hard`

## 15. 将本地修改推送到云端

如果云服务器是通过 Git clone 部署的，推荐用 GitHub 作为中转：

```bash
# 本地提交并推送
git status
git add causal/run_experiment.py deploy/gce/run_causal.sh doc/云服务器部署运行教程.md
git commit -m "Improve causal validation logging and cloud deployment notes"
git push origin sae-re-project
```

然后在云服务器上拉取最新代码：

```bash
cd ~/nlp-re
git fetch origin
git checkout sae-re-project
git pull --ff-only origin sae-re-project
```

拉取后建议先确认入口可用：

```bash
source deploy/gce/common.sh
load_gce_env
source "${VENV_DIR}/bin/activate"
python causal/run_experiment.py --help
```

如果云服务器没有 Git 远端权限，也可以在本地打包后上传：

```powershell
python package_project.py
```

再把生成的 `tar.gz` 上传到云服务器解压覆盖。不要上传 `outputs/`、模型目录、日志目录或本地 wheel 缓存；这些文件体积大，而且会让云端代码状态变得难追踪。
