当前问题：sae的结构性能指标不好，不管是在论文数据集或是我们的数据集上

问题排查：先给你一个判断：**如果论文数据集和你自己的数据集都差，优先怀疑“实现/对齐错误”，而不是先怀疑数据本身。**
因为 Llama Scope 这套 SAE 是严格绑定到 **Llama-3.1-8B-Base**、严格绑定到具体训练位置（R/A/M/TC）和具体 hook 点上的；只要模型版本、hook 位置、归一化、激活函数或评估实现里有一个没对齐，EV 和 Delta LM loss 往往会一起变差。论文里也明确说了，它们的评估是在固定位置插回 SAE 后算 EV 和 Delta LM loss，并且用的是与训练同分布的 held-out 文本。([arXiv][1])

我建议你按这个顺序排查，别乱跳。

**第一层：先排“是不是根本没在评同一个东西”**
最容易出错的是 **模型和位置不一致**。Llama Scope 前台页写得很清楚：它的 checkpoint 是基于 `meta-llama/Llama-3.1-8B` / `Llama-3.1-8B-Base`，并且 `L15R-8x` 这种名字里的 `R` 指的是 **post-MLP residual stream**，不是 resid_pre，也不是 MLP output，更不是 attention output。论文也明确区分了四个训练位置：R、A、M、TC。你如果拿 `L15R-8x` 去挂 `resid_pre`，或者拿 Base 的 SAE 去评 Instruct 且正好又是靠近输出层的位置，指标很容易直接崩。([arXiv][1])

这里最该先做的 sanity check 是：**不要自己训练，先直接跑官方现成 checkpoint 的单层复现。**
比如只测一个公开 checkpoint：`L15R-8x`。它的超参数页直接写了 `hook_point_in = blocks.15.hook_resid_post`、`hook_point_out = blocks.15.hook_resid_post`、`d_model = 4096`、`d_sae = 32768`、`top_k = 50`、`act_fn = jumprelu`。如果你连官方权重都跑不出和论文 Figure 2 同量级的结果，那 90% 以上是你的加载、hook、归一化或评估链路有问题，不是数据问题。([Hugging Face][2])

**第二层：查归一化，這一项特别高危**
Llama Scope 论文专门写了，它们会对输入激活做 **归一化**，让输入进入 SAE 前具有统一的 2-范数；Transcoder 还要同时归一化输入和输出。公开 checkpoint 里也写了 `norm_activation = "dataset-wise"`，例如 `L15R-8x` 还存了 `dataset_average_activation_norm.in = 10.8125`、`out = 10.8125`。所以这里最常见的 bug 有四个：
1）你完全没做它期望的归一化；
2）你做成了 **per-token norm**，而不是 checkpoint 期望的 **dataset-wise norm**；
3）你输入归一化了，但重建后没有正确反归一化就去算 EV / 插回模型；
4）你把 bias、归一化、重建这几个步骤的顺序写错了。
这类错误的典型表现就是：**EV 很低，Delta LM loss 也很差，而且不同层都差得很一致。** ([arXiv][1])

**第三层：查你是不是把“改造后的 TopK SAE”当成了普通 TopK**
Llama Scope 不是“朴素 TopK SAE 直接发布”这么简单。论文说得很明确：它们做了三件事——把 decoder 列向量 2-范数纳入 TopK 计算、训练后把 TopK SAE **后处理成 JumpReLU 变体**、再加上 K-annealing 训练。公开 checkpoint 也显示 `act_fn = "jumprelu"`，并且像 `L15R-8x` 这样的 checkpoint 里还直接存了 `jump_relu_threshold = 0.35546875`。
所以如果你加载官方权重时，前向仍然按“普通 TopK 选 50 个”去跑，而不是按 checkpoint 里的激活函数和阈值去跑，那结果完全可能偏得很厉害。([arXiv][1])

**第四层：查评估实现本身，尤其是 Delta LM loss**
论文里对 Delta LM loss 的定义是：**原模型 loss 与插入 SAE 后模型 loss 的差异**；EV 则是衡量输入方差被重建捕获的比例。评估是在 **5000 万 held-out tokens** 上完成的。这里最容易踩坑的是：

* 你把 Delta LM loss 的符号写反了；
* 原模型和插入 SAE 的模型没有在**同一批 token、同一 mask、同一 reduction 方式**下比较；
* 插入 SAE 时替换了错误张量，或者替换时机不对；
* BOS/EOS 或 padding token 被混进评估，导致方差和 loss 统计都被污染。
  论文还写了：训练时每个文档截断到 **1024 tokens**，前面加 `<beginoftext>`，并且 **BOS/EOS 的激活不参与训练**。你如果评估时把这些特殊 token 大量算进来，EV 很容易发虚，Delta LM loss 也会不稳。([arXiv][1])

**第五层：查训练配置有没有偏离得太多**
如果你是在“复现训练”而不是“只加载公开权重”，那训练超参也是一条大雷。Llama Scope 论文写的是：用 **Adam**，学习率 **8e-4**，前 **10K steps warmup**，最后 **20% steps linearly decay to 0**；同时前 **10% 训练步数**会做 **K-annealing**。它还提到，如果一开始就把激活特征数固定为 K，有些特征会很晚才开始激活，所以才需要 annealing。
如果你没做 warmup / decay、没做 K-annealing、学习率量级差太多，或者优化器实现和它相差很大，常见表现就是：**inactive features 特别多、EV 上不去、L0 不稳定**。([arXiv][1])

**第六层：直接看 feature firing 分布，它能非常快地告诉你哪里不对**
Llama Scope 给了两个很实用的经验阈值：

* **inactive features**：10 万 token 内一次都不触发的特征，理想比例大约 **10%**；
* **ultra-active features**：触发频率大于 0.1 的特征，最好低于 **2%**。
  如果你这两项非常离谱，比如 inactive 一大半，或者 ultra-active 一堆，那往往不是“语义太难学”，而是 **阈值、归一化、学习率、hook 点、数据分布** 里至少有一项没对齐。([arXiv][1])

**第七层：再看你的“自己的数据”问题，但这应该放后面**
只有在前面几层都过了以后，才值得认真怀疑你的数据。因为 Llama Scope 的训练数据是 **SlimPajama**，序列长度 **1024**，而且是大规模同分布训练。你自己的数据如果更小、更窄、更偏领域，确实可能让 feature 更稀、inactive 更多、结构指标更差。
但注意，**这只能解释“自己的数据差”，解释不了“论文数据也差”**。所以只要论文数据也差，就先不要把锅甩给语料规模。([arXiv][1])

你现在最该做的是这个最小闭环：

1. **不要训练。** 先只加载一个官方 checkpoint，比如 `L15R-8x`。
2. 确认模型是 **Llama-3.1-8B-Base**，不是 Instruct。
3. 确认 hook 点就是 `blocks.15.hook_resid_post`。
4. 严格按 checkpoint 里的 `jumprelu + dataset-wise norm` 跑前向。
5. 在一小批文本上同时输出：原激活范数、归一化后范数、重建范数、L0、EV、Delta LM loss。
6. 再画 feature firing 直方图，看 inactive / ultra-active 比例。

如果这一步都不对，那问题基本就锁在这几类里了：

* **hook 点错了**
* **Base / Instruct 用混了**
* **JumpReLU / TopK 用混了**
* **dataset-wise normalization 没对齐**
* **EV 或 Delta LM loss 公式/掩码写错**
* **特殊 token 或 padding 统计进去了**
* **插回模型的位置或替换时机错了**

按经验给你一个概率排序：
**1）hook 与位置不匹配**
**2）归一化不匹配**
**3）把官方 JumpReLU checkpoint 当普通 TopK 跑**
**4）Delta LM loss 实现有 bug**
**5）模型版本 / tokenizer 不一致**
**6）训练调度没对齐**
**7）才轮到你自己的数据问题**
