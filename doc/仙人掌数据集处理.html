CACTUS 数据集替换 Pipeline 文档（用于 SAE-RE 概念发现验证）

你现在要替换现有数据集，改用 CACTUS（Cactus: Towards Psychological Counseling Conversations using Cognitive Behavioral Theory）作为新的数据来源，目标是为 SAE 概念发现任务构造一个小规模、可验证、可复现实验集。

1. 数据集来源

优先使用以下官方来源之一下载：

官方 GitHub

  • Repo: https://github.com/coding-groot/cactus

官方 Hugging Face

  • Dataset: https://huggingface.co/datasets/LangAGI-Lab/cactus

优先选择 Hugging Face 版本进行加载，因为更容易直接读取结构化字段。

推荐加载方式：

from datasets import load_dataset
ds = load_dataset("LangAGI-Lab/cactus", split="train")

2. 任务目标

构造一个用于 SAE 可解释性验证的小规模替换数据集，目标不是复现整套训练，而是先验证以下链路是否跑通：

  1. 从 CACTUS 多轮对话中切分出回合级样本

  2. 构造三类样本：

    • RE
    • NonRE_CBT
    • NonTech_Process

  3. 总样本规模控制在 1500 条

    • 每类 500 条

  4. 使用统一输入模板：

    <client> ... </client>
<therapist> ... </therapist>

  5. 输入整个序列到原模型

  6. 在 blocks.19.hook_resid_post 上取 SAE 编码

  7. 只在 <therapist> ... </therapist> 这一段 token 上聚合 latent activation

  8. 产出可直接用于后续 SAE 分析的 JSONL 文件

3. 关键原则

3.1 最小充分上下文

只使用：

  • 前一轮 client 发言
  • 当前轮 therapist 发言

不要一开始引入：

  • 全对话历史
  • intake_form
  • cbt_plan
  • persona
  • 过长窗口

默认样本单位定义为：

(client_{t-1}, therapist_t)

即：

  • client_prev
  • therapist_curr

3.2 上下文作为条件，不作为读取目标

要把 client 文本作为“判断 therapist 当前句是否体现 RE 技能”的条件信息， 但 最终的 SAE latent 统计只能发生在 therapist span 上。

换句话说：

  • 输入序列包含 client + therapist
  • 但只在 therapist token 范围上做 latent pooling

3.3 第一版不要追求全量准确，只追求跑通

优先构造：

  • 较干净的 RE 样本
  • 较干净的 NonRE_CBT 样本
  • 较干净的 NonTech_Process 样本

第一版允许部分边界例丢弃，不要强行标注混合样本。

4. 数据切分与回合级构造

4.1 原始输入

CACTUS 的一条样本通常包含：

  • dialogue
  • cbt_technique
  • cbt_plan
  • 其他辅助字段

重点使用 dialogue 字段。

4.2 解析 dialogue

需要写一个解析器，把一段多轮对话切成 speaker turn 序列。

目标输出格式类似：

[
  {"speaker": "client", "text": "..."},
  {"speaker": "therapist", "text": "..."},
  {"speaker": "client", "text": "..."},
  {"speaker": "therapist", "text": "..."},
]

4.3 构造回合级 therapist-response 样本

仅保留 therapist turn，配对最近一条 client turn：

{
  "dialogue_id": ...,
  "turn_index": ...,
  "client_prev": "...",
  "therapist_curr": "...",
  "cbt_technique": "...",
  "cbt_plan": "...",
}

4.4 过滤规则

丢弃以下情况：

  • 缺少紧邻前一条 client turn
  • therapist 句子过短（例如少于 5 个词）
  • client 或 therapist 文本明显损坏
  • therapist 当前回合混合多个功能且难以判定主标签
  • 重复样本
  • 明显模板化、完全重复的 boilerplate 句子

5. 三类标签定义

注意：CACTUS 原始数据没有现成的这三类回合级标签，需要你基于 (client_prev, therapist_curr) 自行构造。

5.1 RE

定义： 当前 therapist 回应的主要功能是对 client 刚才表达的内容、情绪、意义、冲突、担忧进行反映、复述、提炼、镜映，而不是推进新的技术动作。

判定标准（满足 2 条即可）：

  1. 明显紧贴前一句 client 内容
  2. therapist 主要在“描述/反映” client 体验
  3. 句子主要功能不是提问、建议、教育、任务推进

典型例子：

  • “听起来你一到那个场景，就会马上觉得自己会被否定。”
  • “你其实不是不想尝试，而是很怕失败之后更失望。”

保留：

  • simple reflection
  • complex reflection
  • feeling reflection
  • meaning reflection

剔除：

  • reflection + question 混合句
  • reflection + CBT intervention 混合句

5.2 NonRE_CBT

定义： 当前 therapist 回应的主要功能是推进 CBT 技术动作，而不是反映 client。

包括但不限于：

  • evidence checking
  • reality testing
  • reframe / cognitive restructuring
  • psychoeducation
  • guided discovery
  • behavioral planning
  • homework suggestion
  • decatastrophizing
  • alternative perspective

典型例子：

  • “我们来看看有哪些证据支持这个想法，又有哪些证据不支持它。”
  • “你愿不愿意这周做个小实验，记录一下当时的自动想法？”

必须排除：

  • 只是一般性提问但没有 CBT 技术意图的句子
  • 明显的流程话语
  • 混合 RE + 技术推进的句子

5.3 NonTech_Process

定义： 当前 therapist 回应不是 RE，也不是核心 CBT 技术动作，而是流程管理、会谈组织、寒暄、过渡、一般支持或信息性探查。

包括：

  • greeting
  • naming / self-introduction
  • opening / closing
  • transition
  • session management
  • general support
  • non-technical information-gathering question

典型例子：

  • “在开始之前，我先确认一下你希望我怎么称呼你。”
  • “谢谢你今天愿意分享这些。”
  • “我们今天先聊到这里，下次再继续。”

注意：

  • 此类样本不要过度使用疑问句模板
  • 避免让整个负类退化成 question detection

6. 抽样目标

最终只保留总计 1500 条样本：

  • RE: 500
  • NonRE_CBT: 500
  • NonTech_Process: 500

6.1 抽样优先级

优先保证“纯度”，其次才是“覆盖面”。

RE 组要求

优先选择：

  • therapist 功能单一
  • 高度依赖 client_prev
  • 不包含 question 尾巴
  • 不包含技术推进动作
  • 不是总结式混合句

NonRE_CBT 组要求

优先选择：

  • 技术意图明显
  • 不是单纯一般问句
  • 不是支持性寒暄
  • 尽量覆盖多个 CBT 子类型，不要只集中一种

NonTech_Process 组要求

优先选择：

  • 流程/寒暄/一般支持/过渡句
  • 少量信息收集问题可以保留
  • 不要让 “问句” 占比过高
  • 不要混入实际 CBT 技术动作

6.2 去重与平衡

必须去重：

  • 近似重复 therapist 句子
  • 模板化几乎相同的 response

建议额外统计：

  • 平均 therapist token 长度
  • 问句占比
  • 不同 cbt_technique 来源分布
  • 不同对话来源分布

尽量避免某一类只来自极少数技术或少量 dialogue。

7. 输入模板

每条最终样本必须被格式化成：

<client>
{client_prev}
</client>
<therapist>
{therapist_curr}
</therapist>

不要使用：

  • [CLIENT]
  • [THERAPIST]
  • chat template role token
  • system prompt

第一版固定使用上述 XML 风格模板。

8. Token span 对齐要求（非常重要）

后续 SAE 分析只在 therapist span 上聚合 latent activation，因此你必须在数据预处理中保留 therapist token 的准确边界。

必须输出：

  • formatted_text
  • therapist_text
  • client_text
  • therapist_char_start
  • therapist_char_end

如果后续 tokenizer 可直接返回 offset mapping，则还应支持推导：

  • therapist_token_start
  • therapist_token_end

建议做法：

  1. 先生成完整 formatted_text
  2. 用字符串匹配定位 <therapist>\n 后 therapist 内容在完整字符串中的字符区间
  3. 后续模型 tokenize 时用 offset mapping 找到 therapist 对应 token span

9. 后续 SAE 读取规范

固定要求

  • 输入整个 formatted_text 到原模型
  • 在 blocks.19.hook_resid_post 读取残差流激活
  • 用已加载的 Llama Scope SAE 对该层激活进行编码
  • 只对 therapist span 范围内 token 的 SAE feature activation 做聚合

第一版需要同时支持的 pooling

不要只做 max，至少同时实现：

  • mean
  • sum
  • max

输出时分别保存：

  • latent_mean
  • latent_sum
  • latent_max

如果内存太大，可只保存 therapist span 上非零 feature 统计，或按样本逐条在线计算。

10. 推荐输出文件

10.1 标注后的主数据文件

cactus_re_small_1500.jsonl

每行格式建议如下：

{
  "sample_id": "cactus_000123_t08",
  "dialogue_id": "cactus_000123",
  "turn_index": 8,
  "label": "RE",
  "subtype": "meaning_reflection",
  "client_prev": "...",
  "therapist_curr": "...",
  "formatted_text": "<client>\n...\n</client>\n<therapist>\n...\n</therapist>",
  "therapist_char_start": 42,
  "therapist_char_end": 118,

  "source_cbt_plan": "..."
}

10.2 统计报告

cactus_re_small_1500_stats.json

至少包含：

  • 每类样本数
  • 每类平均长度
  • 问句占比
  • 不同来源 technique 分布
  • 重复率检查结果

10.3 可人工抽查文件

字段建议：

  • sample_id
  • label
  • client_prev
  • therapist_curr
  • source_cbt_technique

11. 推荐实现策略

11.1 第一阶段：规则筛选

先用启发式规则拿到较干净的候选池：

RE 候选规则（示例）

匹配 therapist_curr 中的反映式模式：

  • 听起来
  • 你觉得
  • 你会觉得
  • 好像
  • 对你来说
  • 你其实是在
  • 你担心的是

同时排除：

  • 结尾带明显问号的句子
  • 包含“我们来看看”“我们试试”“这周你可以”等技术推进短语

NonRE_CBT 候选规则（示例）

匹配：

  • 我们来看看
  • 有哪些证据
  • 有没有另一种解释
  • 这周你可以
  • 记录一下
  • 练习
  • 实验
  • 重构
  • 重新看待
  • 挑战这个想法

NonTech_Process 候选规则（示例）

匹配：

  • 你好
  • 怎么称呼
  • 谢谢你分享
  • 今天先到这里
  • 下次我们继续
  • 我想先了解
  • 能和我说说

11.2 第二阶段：人工或 LLM 复核

对规则筛出的候选池做二次复核，目标是保证 1500 条最终样本纯度够高。

如果实现 LLM 复核，输出仍然只能保留最终三类之一：

  • RE
  • NonRE_CBT
  • NonTech_Process

不确定样本直接丢弃。

12. 质量门槛

只有满足以下条件，才算 pipeline 通过：

  1. 成功下载并加载 CACTUS
  2. 成功解析多轮 dialogue 为回合级 turn
  3. 成功构造 1500 条样本
  4. 三类各 500 条
  5. 每条有完整 formatted_text
  6. 每条能正确定位 therapist span
  7. 后续程序能把 formatted_text 输入模型
  8. 能在 blocks.19.hook_resid_post 取到激活
  9. 能只对 therapist span 聚合 latent activation
  10. 至少能输出一版 mean / sum / max pooling 结果

13. 额外提醒

不要做的事

  • 不要把整段对话全部作为一个样本
  • 不要把 client token 与 therapist token 混在一起聚合 latent
  • 不要把 CACTUS 的 cbt_technique 直接当作 RE 标签
  • 不要把所有疑问句都塞进 NonTech_Process
  • 不要引入过长上下文窗口
  • 不要在第一版加入复杂的多轮跨句推理

第一版的唯一目标

先验证：

  • CACTUS 是否能被切成三类小规模样本
  • therapist-span 限定的 SAE 读取链路是否能稳定跑通
